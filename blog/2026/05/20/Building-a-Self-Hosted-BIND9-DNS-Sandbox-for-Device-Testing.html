Building a Self-Hosted BIND9 DNS Sandbox for Device Testing

Posted on May 20, 2026 ()

Categories: dns  networking  testing 

A hardware manufacturer that makes security and surveillance equipment came to us with a testing problem.

Their devices ship worldwide, and on first power-up each one contacts a registration endpoint to activate itself and pull configuration. Which endpoint depends on the destination market — there is a different regional service for each, and several of them are geo-restricted, reachable only from within the region they serve.

That makes pre-shipment testing awkward. Before a batch goes out, they want to confirm each device actually completes its registration handshake. But from the factory they cannot reach most of the registration servers, and even where they can, hammering a production activation service with test traffic is not something anybody wants.

What they needed was a way to make devices believe they were talking to a registration server while actually talking to something under their control — without modifying device firmware, since the whole point is to test the shipping build.

The answer is DNS. Devices resolve a hostname before they connect to it, so whoever controls the answer controls the destination.

The approach

A single BIND9 server that:

  • answers authoritatively for the specific domains being tested, pointing them at test endpoints;
  • forwards everything else to real upstream resolvers, so normal internet access still works;
  • is pointed at by devices via DHCP or a static resolver setting.

That second point is what makes this practical. A test environment that breaks all other name resolution is a poor test environment — devices frequently need NTP, certificate validation and firmware checks to get far enough into their startup sequence for registration to be worth testing. Everything except the domains you deliberately override should behave completely normally.

  Device
    │  DNS query
    ▼
  ┌──────────────────────────────────┐
  │ BIND9 test resolver              │
  │                                  │
  │  test domains ──► local zone     │──► test endpoint
  │  everything else ──► forwarders  │──► real internet
  └──────────────────────────────────┘

Setting it up

A small cloud VM running Ubuntu 22.04 is plenty. DNS is not demanding.

1. Install

$ sudo apt update
$ sudo apt install -y bind9 bind9utils dnsutils

2. Configure recursion and forwarding

In /etc/bind/named.conf.options:

options {
    directory "/var/cache/bind";

    recursion yes;
    allow-query { 10.0.0.0/8; 192.168.0.0/16; };

    forwarders {
        8.8.8.8;
        1.1.1.1;
    };
    forward only;

    dnssec-validation no;

    listen-on { any; };
    listen-on-v6 { none; };
};

Two settings deserve comment.

allow-query must be restricted to your own networks. An open recursive resolver on the public internet will be found and abused — within hours, typically — as an amplifier for reflection attacks. This is not a theoretical risk; it is the single most important line in the file.

dnssec-validation no is deliberate here and would be wrong almost anywhere else. We are about to return forged answers for domains we do not own, which is precisely what DNSSEC exists to prevent. If the real domains are signed, validation will correctly reject our fabricated records. Turning it off is acceptable only because this is an isolated test environment — and it is a good reminder that this technique works only against names that are not DNSSEC-validated by the client.

3. Define a zone for the domains under test

In /etc/bind/named.conf.local:

zone "registration.example.com" {
    type master;
    file "/etc/bind/db.registration.example.com";
};

And the zone file:

$TTL 60
@   IN  SOA ns1.registration.example.com. admin.registration.example.com. (
        2026052001  ; Serial - bump on every edit
        3600        ; Refresh
        600         ; Retry
        86400       ; Expire
        60 )        ; Negative cache TTL

@       IN  NS   ns1.registration.example.com.
ns1     IN  A    10.0.1.53

@       IN  A    10.0.1.100
api     IN  A    10.0.1.100
device  IN  A    10.0.1.100
*       IN  A    10.0.1.100

Notes on choices here:

  • A 60-second TTL. Deliberately short. When you repoint a test domain you want devices picking up the change in a minute, not in a day. Cache lifetime is a tuning parameter, and in a test environment short beats efficient.
  • The wildcard record. Devices often use hostnames you did not anticipate, and firmware rarely documents every endpoint it contacts. A wildcard catches them all and saves a debugging cycle for each one you would otherwise discover the hard way.
  • The serial number. Written as a date plus a counter. BIND refuses to reload a zone whose serial has not increased, and forgetting to bump it is the most common reason a change appears to have no effect.

4. Check and reload

$ sudo named-checkconf
$ sudo named-checkzone registration.example.com \
       /etc/bind/db.registration.example.com
$ sudo systemctl restart bind9

Use the checking tools before every restart. A syntax error leaves BIND refusing to start, and since everything in the test environment depends on it, that is a fast way to make it look as though every device has failed at once.

5. Open the firewall

DNS needs both protocols — UDP for ordinary queries, TCP for responses too large for a UDP packet and for zone transfers:

$ sudo ufw allow from 10.0.0.0/8 to any port 53 proto udp
$ sudo ufw allow from 10.0.0.0/8 to any port 53 proto tcp

On a cloud VM the security group needs the same rules, sourced from your test network only. Opening UDP/53 to 0.0.0.0/0 recreates the open-resolver problem regardless of what allow-query says.

People forget TCP/53 constantly. The symptom — most lookups work, but a few specific ones fail — is confusing enough to burn an afternoon.

6. Point devices at it

For a device you can log into:

nameserver 10.0.1.53

For appliances you cannot configure directly — which is most of them — set the DNS server in the DHCP scope for the test network. Every device that boots there picks it up automatically, which is both less work and a better test, since it exercises the same configuration path a customer's network would.

Verifying it works

$ dig @10.0.1.53 api.registration.example.com +short
10.0.1.100

$ dig @10.0.1.53 www.google.com +short
142.250.x.x

The first confirms the override, the second confirms forwarding still works. Both need to pass — an environment where the override works but general resolution is broken will produce device failures that have nothing to do with what you are testing.

From the device side, tcpdump -i any -n port 53 on the resolver shows exactly what each device asks for as it boots. This is more useful than it sounds: it is effectively free documentation of a device's network dependencies, including ones the datasheet never mentions.

Adding a domain later

The routine, once it is running:

  1. Edit the zone file (or add a new zone for a new domain).
  2. Bump the serial.
  3. sudo named-checkzone <zone> <file>
  4. sudo rndc reload <zone>

rndc reload re-reads a single zone without restarting the daemon or dropping the cache — a few milliseconds rather than a service interruption. On a shared test environment that difference matters.

Things worth knowing before you build this

HTTPS will fail, and that is expected. DNS controls where the connection goes, not what certificate is presented. A device connecting to your test endpoint over TLS will get a certificate that does not match the hostname and, if it validates properly, refuse to continue. Depending on what you are testing, you either need a certificate for the real hostname on your test endpoint, or a device build that trusts your test CA, or you accept that you are testing the resolution and connection path rather than the full handshake. Decide which before you start, because discovering it halfway through is dispiriting.

Devices that ignore DHCP DNS. A surprising number of embedded devices have a hard-coded resolver — 8.8.8.8 is the usual choice — and never consult what DHCP gave them. For those, a DNAT rule on the test network gateway redirecting all outbound port 53 to your resolver is the workaround:

$ sudo iptables -t nat -A PREROUTING -i eth1 -p udp --dport 53 \
       -j DNAT --to-destination 10.0.1.53

This also catches devices using DNS-over-TLS on port 853 if you extend it, though DNS-over-HTTPS on port 443 is genuinely hard to intercept this way and may require blocking the known DoH endpoints outright.

Keep it isolated. A resolver that lies about real domains must not be reachable from anything but the test network. Separate VPC or subnet, no route to production, restrictive security group. The failure mode — a production system silently resolving a real domain to a test endpoint — is the kind of problem that takes a long time to diagnose because nobody suspects DNS.

Response Policy Zones are the alternative. BIND's RPZ feature can rewrite answers for arbitrary domains without hosting a full zone for each, which scales better if you are overriding many domains across many parents. For a handful of test domains, plain authoritative zones are simpler to reason about and easier to hand to a colleague.

Why this generalises

The customer's use case was device registration, but the pattern applies far more widely:

  • Testing against unreachable services — geo-restricted, partner-operated, or not built yet.
  • Failure injection — point a dependency at an endpoint that returns errors, times out, or responds very slowly, and see what your software does.
  • Migration rehearsal — verify a system works against a new backend before changing any real DNS records.
  • Air-gapped environments — provide plausible answers for external services that are not reachable at all.

In every case the appeal is the same: you redirect a dependency without touching the thing under test. No firmware rebuild, no configuration change, no code modification. The system behaves exactly as it will in production, and only its view of the network has changed — which is a considerably stronger test than one where you had to modify the software to make it testable.
