Multiple Public IPs on One VM: Why You Need Policy Routing

Posted on May 13, 2026 ()

Categories: networking  linux  routing 

A customer attached several elastic network interfaces to one virtual machine and bound a separate public IP to each. The topology in the cloud console looked exactly right. Only the first public IP worked. The others answered nothing — no SSH, no HTTP, no ping.

Nothing was misconfigured on the cloud side. The problem is a property of how Linux routes packets, and it catches people out reliably enough that it is worth explaining properly rather than just pasting the fix.

The setup

A typical multi-IP arrangement:

InterfaceRolePrivate IPPublic IP
eth0Primary ENI10.0.1.10IP-1
eth1Secondary ENI10.0.1.11IP-2
eth2Secondary ENI10.0.1.12IP-3

Every interface is up. Every interface has an address. Every public IP is bound in the console. Traffic to IP-1 works perfectly; traffic to IP-2 and IP-3 disappears.

What actually happens to the packet

Follow a single request to IP-2 through the system.

Inbound is fine. The packet arrives from the internet, the cloud network translates IP-2 to 10.0.1.11, and it is delivered to eth1. The kernel accepts it and hands it to the listening socket. Everything works as expected.

Outbound is where it falls apart. The application writes a reply. The kernel now has to decide which interface to send it out of, and it makes that decision by consulting the routing table:

$ ip route show
default via 10.0.1.1 dev eth0
10.0.1.0/24 dev eth0 proto kernel scope link src 10.0.1.10
10.0.1.0/24 dev eth1 proto kernel scope link src 10.0.1.11
10.0.1.0/24 dev eth2 proto kernel scope link src 10.0.1.12

There is one default route, and it points at eth0. The destination is somewhere on the internet, no more specific route matches, so the reply goes out via eth0 — regardless of which interface the request arrived on.

The reply now leaves the VM on the wrong interface, and the cloud network drops it. Every cloud platform performs a source/destination check: a packet leaving eth0 must carry eth0's source address. A packet with 10.0.1.11 as its source, egressing from the interface that owns 10.0.1.10, is discarded as spoofed.

This is asymmetric routing — in one interface, out another. On a physical network with no filtering it sometimes limps along. In a cloud VPC it never does, and that is deliberate: those anti-spoofing checks are what keep tenants from forging each other's addresses.

The result is that the client sees a connection that opens and then goes silent, or simply times out. From inside the VM, tcpdump shows the request arriving and the reply being sent, which is exactly the kind of evidence that sends people looking at security groups for an hour. The packet really is leaving; it is just leaving from the wrong door.

The fix: one routing table per interface

The underlying limitation is that a single routing table can only express "packets to destination X go out interface Y". What we need is "packets from source A go out interface Y" — a decision based on the source address, which ordinary routing does not consider at all.

Linux policy routing solves this. It supports many routing tables plus a set of rules that select which table to consult, and those rules can match on source address. So we give every interface its own table containing its own default route, and add a rule sending traffic from each source IP to the matching table.

Step 1: a routing table per interface

Name the tables in /etc/iproute2/rt_tables:

$ echo "101 eni1" | sudo tee -a /etc/iproute2/rt_tables
$ echo "102 eni2" | sudo tee -a /etc/iproute2/rt_tables

Then give each one a default route out of its own interface:

$ sudo ip route add default via 10.0.1.1 dev eth1 table eni1
$ sudo ip route add default via 10.0.1.1 dev eth2 table eni2

Adding a subnet route to each table as well avoids surprises with local traffic:

$ sudo ip route add 10.0.1.0/24 dev eth1 src 10.0.1.11 table eni1
$ sudo ip route add 10.0.1.0/24 dev eth2 src 10.0.1.12 table eni2

Step 2: rules that select the table by source address

$ sudo ip rule add from 10.0.1.11 lookup eni1
$ sudo ip rule add from 10.0.1.12 lookup eni2

These say: any packet whose source address is 10.0.1.11 is routed using table eni1, whose default route exits via eth1.

Note that the rules match the private addresses, not the public ones. The VM never sees its public IPs — the cloud network handles that translation outside the guest. Trying to match on the public IP is a common early mistake and produces rules that never fire.

eth0 needs no rule. Anything not matched by a rule falls through to the main table, whose default route already points at eth0.

Step 3: verify

$ ip rule show
0:      from all lookup local
32764:  from 10.0.1.12 lookup eni2
32765:  from 10.0.1.11 lookup eni1
32766:  from all lookup main
32767:  from all lookup default

$ ip route show table eni1
default via 10.0.1.1 dev eth1
10.0.1.0/24 dev eth1 scope link src 10.0.1.11

Rules are evaluated in ascending priority order, so the specific source rules are consulted before the catch-all main lookup. That ordering is what makes the whole thing work.

To test a route decision without generating traffic:

$ ip route get 8.8.8.8 from 10.0.1.11
8.8.8.8 from 10.0.1.11 via 10.0.1.1 dev eth1 table eni1

If that says dev eth1, the kernel will do the right thing. If it says dev eth0, a rule is missing or misordered.

Step 4: make it survive a reboot

Everything above is runtime state and vanishes on reboot. This is worth being deliberate about, because the failure mode — works perfectly for weeks, then breaks during an unrelated maintenance restart — is unpleasant to debug.

With netplan:

network:
  version: 2
  ethernets:
    eth1:
      addresses: [10.0.1.11/24]
      routes:
        - to: 0.0.0.0/0
          via: 10.0.1.1
          table: 101
      routing-policy:
        - from: 10.0.1.11
          table: 101

With systemd-networkd, use [Route] and [RoutingPolicyRule] sections in the .network file for each interface. On NetworkManager systems, nmcli supports ipv4.routes and ipv4.routing-rules per connection.

Before and after

Before — one default route, everything exits eth0:

  Internet
     │  request to IP-2
     ▼
  ┌──────────────────────────────┐
  │ VM                           │
  │  eth0 (10.0.1.10) ◄── default route
  │  eth1 (10.0.1.11) ◄── request lands here
  │  eth2 (10.0.1.12)            │
  └──────────────────────────────┘
     │  reply, src 10.0.1.11, out via eth0
     ▼
   dropped by source check   ✗

After — each source address has its own table:

  Internet
     │  request to IP-2
     ▼
  ┌──────────────────────────────┐
  │ VM (policy routing)          │
  │  eth0 + main table           │
  │  eth1 + table eni1  ◄── rule: from 10.0.1.11
  │  eth2 + table eni2  ◄── rule: from 10.0.1.12
  └──────────────────────────────┘
     │  reply, src 10.0.1.11, out via eth1
     ▼
   delivered   ✓

Notes from practice

  • Applications must bind to the right address. Policy routing fixes replies on established connections, because those already carry the correct source. For connections the VM initiates, the source address is chosen when the socket is created — so a program that binds to 0.0.0.0 will get eth0's address and route out eth0. If you want outbound traffic on a specific interface, bind explicitly.
  • Reverse path filtering. Some distributions ship rp_filter in strict mode, which drops inbound packets whose source would not route back out the same interface. Once policy routing is correct this usually resolves itself, but if things still fail, loose mode (net.ipv4.conf.all.rp_filter = 2) is the usual remedy.
  • Same-subnet interfaces are the hard case. When every ENI is in one subnet, ordinary routing genuinely cannot distinguish them — the routes are identical apart from the interface. Different subnets per ENI sometimes appear to work without policy routing, which makes the same-subnet case feel like a bug when it is not.
  • The pattern generalises. This is the same technique used for multi-homed servers, dual-ISP routers with per-source egress, and VPN split routing. Any time the routing decision depends on where a packet came from rather than where it is going, ip rule is the tool.

The thing I would most want a reader to keep: when traffic arrives but replies vanish, stop looking at firewalls and look at which interface the reply leaves by. ip route get <dest> from <src> answers that question in one line.
