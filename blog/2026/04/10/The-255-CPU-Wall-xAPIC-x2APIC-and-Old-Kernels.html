The 255-CPU Wall: xAPIC, x2APIC and Old Kernels

Posted on April 10, 2026 ()

Categories: linux kernel  hardware  troubleshooting 

A customer wanted to move a database server onto bigger hardware. The old instance had 64 virtual CPUs and 128 GB of memory; business growth meant they needed several times that. The target was a modern AMD EPYC instance with 384 vCPUs and just over a terabyte of RAM.

It would not boot. The console showed the kernel starting normally, then a wall of soft lockup warnings, then nothing.

The complication was the operating system. The customer's image ran a vendor-maintained enterprise Linux distribution on kernel 3.10.107 — a 2017 kernel, in an image built in early 2022, on a distribution release that had since reached end of life. The OS vendor's team declined to help on the grounds that the release was no longer supported, which is a fair position and also not much use when a production database needs to move next week.

So we investigated it ourselves. The answer turned out to be an 8-bit field specified decades ago.

Establishing what actually breaks

Before theorising it was worth finding the boundary. We got the image into a lab environment — a 32 TB image copied across regions, which by itself took about a week and is a lesson in its own right — and booted it on everything we could get our hands on.

The pattern is unusually clean for a kernel bug. It is not "AMD is broken" — AMD Milan with 232 vCPUs was fine. It is not "large machines are broken" — Intel at 224 vCPUs was fine, and we could not test Intel above 224 because no such instance existed at the time. It is specifically AMD at 256 vCPUs or more.

A threshold at 256 in a computer is never a coincidence. It is 28. Something in the boot path is using an 8-bit identifier, and at 256 CPUs it runs out of values.

The kernel messages fit that reading. The lockups were reported against a specific CPU that never came up, along with RCU stall warnings — the read-copy-update subsystem complaining that a grace period could not complete because some CPU had not responded. A CPU that is online as far as the scheduler is concerned but cannot be reached is exactly what you would expect if the mechanism used to interrupt it cannot address it.

The 8-bit field: xAPIC

Every CPU in an x86 system has a local APIC — Advanced Programmable Interrupt Controller — and every local APIC has an ID. That ID is how one CPU sends an interrupt to another. It is used constantly: waking a CPU during boot, IPIs for TLB shootdowns, timer delivery, rescheduling requests.

In the original xAPIC design, that ID field is 8 bits wide, and one value is reserved for broadcast. That gives you 255 addressable CPUs, and the APIC registers are reached through memory-mapped I/O.

When that was specified, 255 CPUs was an absurd number for a single machine. It is not absurd any more.

The successor, x2APIC, widens the ID to 32 bits — roughly four billion possible CPUs, comfortably beyond anything on the horizon — and switches from MMIO to model-specific registers, which is also faster since it avoids a memory transaction for every access. The catch is that enabling x2APIC generally requires interrupt remapping through the IOMMU, so the platform, firmware and kernel all have to cooperate.

You can check what your own machine settled on:

$ dmesg | grep -i x2apic
$ grep -c ^processor /proc/cpuinfo

On a modern kernel and modern hardware you will typically see x2APIC enabled. That is the healthy case.

Why Intel was fine and AMD was not

This is where the vendor split stops being mysterious. Intel introduced x2APIC around 2008, and kernel support for Intel's implementation is correspondingly old and well-exercised — by 2013, when the 3.10 series was released, that path had been in the tree and in production for years.

AMD came to x2APIC considerably later, with the EPYC 7002 family in 2019 — six years after kernel 3.10 was released. A 2013 kernel cannot be expected to correctly enable a platform feature that did not exist on that vendor's hardware when it was written.

So on an AMD machine the 3.10 kernel does not enable x2APIC. It falls back to xAPIC, which it handles perfectly well — and which can only address 255 CPUs. Boot proceeds, the kernel brings up processors one by one, and somewhere past the 255th it tries to send a startup IPI to a CPU whose APIC ID does not fit in the field. The interrupt goes nowhere. The boot CPU waits. RCU notices a CPU that never reports a quiescent state. The watchdog fires soft lockup warnings. The machine wedges.

Note that nothing here is a *crash*. There is no panic, no oops, no error message naming the real problem. The kernel is behaving exactly as designed, using a mechanism that is simply too small for the machine it has been given. That is why the symptom is a hang rather than a diagnostic — and why the CPU count threshold, not the error text, is the thing that identifies it.

The upstream fix

The relevant upstream work is a 2020 patch by David Woodhouse, "x86/apic: Fix x2apic enablement without interrupt remapping" (commit 26573a97746c), which was backported into the stable series and landed in Linux 5.4.87.

The vendor kernel line the customer would have needed picked it up in a 5.4.87-based release in early 2021, whose changelog lists AMD 256-core support as an explicit feature. That is a useful confirmation of the boundary: no kernel earlier than 5.4.87 in that line supports 256 or more AMD cores, and 3.10.107 is a very long way earlier.

Which closed off the obvious fix. The distribution release the customer was running topped out at kernel 4.14. The fix needs 5.4.87 or later. There is no upgrade path within that distribution — you would have to change distribution, which for a production database image with a large amount of accumulated customisation is a project, not a maintenance window.

What we actually did

With the software fix unavailable on the customer's timeline, the options were both about staying under 256.

Option A — disable hyper-threading. Turning SMT off on the 256-vCPU AMD instance halves it to 128 vCPUs, comfortably under the limit, while keeping all of the memory. You keep the large memory footprint that a database wants, and you throw away half your threads to do it. For a workload that is memory-bound rather than CPU-bound this can genuinely be the right trade.

Option B — a 192-vCPU instance. A different size in the same modern AMD family: 192 vCPUs, 576 GB of memory. 192 is under 255, so the problem never arises, and every thread is available. The cost is less memory than the largest instance offers.

The customer chose Option B. It tripled their CPU capacity — 64 vCPUs to 192 — on a proven-compatible configuration, with no OS migration and no memory-bandwidth compromise from disabled SMT. The upgrade went through without incident.

It is worth being clear that this is a workaround, not a fix. They are still running a 2017 kernel on an end-of-life distribution, and the next capacity increase will hit the same wall. The real fix is an OS upgrade, and this case gave them a concrete reason to schedule it.

What to take away

  • 255 vCPUs is a real boundary, not a round number. If you are crossing it — in either direction, on any platform — check your kernel version first. Everything below it can look perfectly healthy right up to the moment it is not.
  • Old kernels fail silently on new hardware. There was no message saying "too many CPUs for xAPIC". The kernel did something reasonable with an obsolete mechanism and hung. Expect vintage kernels on new silicon to produce hangs rather than diagnostics.
  • Feature support is per-vendor and per-generation. "The kernel supports x2APIC" was true for Intel years before it was true for AMD. Generalising from one vendor to the other is how you get surprised.
  • End-of-life means end of options. The technical fix existed and had existed for years. It was unreachable because the distribution had no path to a kernel new enough to contain it. OS lifecycle is a capacity-planning constraint, not just a security one.
  • Test the instance type change before you need it. Most of the elapsed time here went into logistics — a week waiting for hardware, another week copying a 32 TB image between regions — not into diagnosis. A short test boot on the target shape, done early, would have surfaced this before it was urgent.
