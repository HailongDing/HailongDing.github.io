Recovering a Disk After sgdisk Was Run on the Wrong Device

Posted on April 7, 2026 ()

Categories: storage  recovery  linux 

A customer's data disk had filled up. They expanded the cloud block storage volume past 2 TiB, and then discovered that the partition table on it was MBR — which cannot describe a partition larger than 2 TiB. The correct next step is to convert the disk to GPT, and sgdisk -g does exactly that.

They ran:

$ sgdisk -g /dev/vdb1

instead of:

$ sgdisk -g /dev/vdb

One character. The command completed without complaint, and afterwards df reported the filesystem's size as roughly 64 zettabytes — the standard sign that a size field has been read as garbage and interpreted as a very large unsigned number. Two terabytes of production data was inaccessible.

The good news, and the reason this post exists, is that all of it came back. Here is what went wrong and exactly how it was recovered.

Why that one character matters

/dev/vdb is a device. /dev/vdb1 is a partition on that device — a window onto a range of blocks inside it.

Partition tables live at the start of a device, because that is where firmware and the kernel look for them. When you tell sgdisk to write a GPT to /dev/vdb1, it does not refuse on the grounds that a partition is not a disk. It has been handed a block device and it writes a partition table to the beginning of it, exactly as instructed.

The beginning of /dev/vdb1, however, is not free space. It is where the ext4 primary superblock lives — the structure describing block size, block count, inode count and the location of everything else. GPT writes a protective MBR in the first 512 bytes and its own header in the next 512. Those 1024 bytes landed directly on top of the filesystem's most important metadata.

Hence 64 zettabytes: the kernel read a block count out of what was now GPT header data and did the arithmetic faithfully.

The critical point for recovery is what was not touched. Only the first 1024 bytes of the partition were overwritten. The file data, the directory structure, the inode tables and the block bitmaps — everything past that first kilobyte — was untouched. This was a metadata accident, not a data accident, and ext4 keeps backup copies of exactly the metadata that was lost.

MBR versus GPT, briefly

MBRGPT
Maximum partition size2 TiBEffectively unlimited
Partition table copiesOne, at the startPrimary at start, backup at end
Integrity checkingNoneCRC32 on headers and entries

The 2 TiB limit is not arbitrary: MBR stores the starting sector and length as 32-bit values, and 232 sectors of 512 bytes is 2 TiB. Any time you grow a volume past that mark, MBR stops being an option.

The recovery

A warning before the commands: take a snapshot of the volume first. Every step below writes to the disk, and if something goes wrong mid-way you want the option to start over. Cloud block storage makes this cheap and it takes a minute. It is the single highest-value action available in a situation like this, and it is worth taking even though — perhaps especially because — you are under pressure to just fix it.

Step 0: stop the disk from breaking the boot

Comment out the data disk's entry in /etc/fstab:

# /dev/vdb1  /data  ext4  defaults  0  2

A corrupted filesystem in fstab can cause the boot to fail or drop into emergency mode, which turns a data problem into an availability problem while you are still working out what happened. Do this before any reboot.

Step 1: find the backup superblocks

$ sudo mke2fs -n /dev/vdb1

This looks alarming, and it is worth being precise about why it is safe. The -n flag means dry run: mke2fs calculates the layout it would create and prints it, without writing anything. Because superblock backup placement is deterministic — derived from block size and filesystem geometry — the offsets it prints for a filesystem it would create with the same parameters are the offsets your existing backups are actually at.

The output ends with a list like:

Superblock backups stored on blocks:
    32768, 98304, 163840, 229376, 294912, 819200, 884736, ...

Do not omit the -n. Without it this command formats the disk.

Step 2: erase the GPT header written onto the partition

$ sudo dd if=/dev/zero of=/dev/vdb1 bs=512 count=2
$ sudo partprobe /dev/vdb

This zeroes the first 1024 bytes of the partition — the protective MBR and GPT header that should never have been there. We are not destroying filesystem data; that space is already ruined, and clearing it prevents anything from trying to interpret it as a valid partition table later.

Check of= twice before pressing return. of=/dev/vdb here would wipe the real partition table.

Step 3: restore the superblock from a backup

$ sudo e2fsck -fy -b 32768 /dev/vdb1

-b 32768 tells e2fsck to ignore the primary superblock and use the backup at block 32768 instead. Having a consistent superblock to work from, it rebuilds the primary one and reconciles the filesystem against it. -f forces the check even though the filesystem believes itself clean, and -y answers yes to the questions it will ask.

This is the moment the filesystem becomes describable again. If block 32768 does not work, try the next offset from the list in step 1.

Step 4: do the conversion properly, and grow the partition

$ sudo sgdisk -g /dev/vdb
$ sudo growpart /dev/vdb 1

The original goal, on the device this time. sgdisk -g converts the disk to GPT, and growpart extends partition 1 to fill the newly available space.

Step 5: full filesystem check

$ tmux new -s recovery
$ sudo e2fsck -fyy -C 0 /dev/vdb1

Now check the whole filesystem properly. On a multi-terabyte volume this runs for a long time — hours is normal — so start it inside tmux or screen. If your SSH session drops during an unprotected e2fsck, the process dies with it, potentially mid-write. -C 0 prints a progress bar, which matters mostly for your own sanity.

The doubled -yy answers yes to every prompt including the ones -y alone will still stop on.

Step 6: resize the filesystem into the larger partition

$ sudo resize2fs /dev/vdb1

The partition is now bigger, but the filesystem inside it still describes its old size. With no size argument, resize2fs grows it to fill the partition.

Step 7: mount somewhere harmless and verify

$ sudo mount /dev/vdb1 /mnt/recovery_test
$ df -TH

Mount to a scratch directory rather than the production path, so you can inspect the result without anything starting to write to it. Confirm the size is sensible, spot-check some files, then restore the fstab entry and mount properly.

The final state: 3.2 TB total, 2.1 TB used at 69% — every byte of the customer's data present and readable.

Why this worked

Recovery was possible because of a design decision made in the ext filesystem family a long time ago: the superblock is replicated across the disk. The primary copy at the start is the one that gets used, and it is also the one most exposed to accidents at the start of the device — bad partition commands, stray dd, a bootloader install pointed at the wrong target. The backups sit at predictable intervals deep inside the filesystem where almost nothing writes by accident.

The chain of reasoning that made the recovery straightforward:

  • Only the first 1024 bytes were overwritten.
  • Those bytes held metadata, not file contents.
  • ext4 keeps copies of that metadata at known offsets.
  • Therefore restore the metadata and the data becomes reachable again.

None of this would help against a real data-destroying accident — mkfs on a live filesystem, or dd across a large range. The reason a corrupted partition table so often looks like total loss and so often is not is that partition tables are tiny and the data is enormous.

Lessons

  • Snapshot before touching partition tables. It costs a minute. This entire incident would have been a five-minute rollback.
  • Device or partition — read the argument out loud. sgdisk, parted and fdisk operate on devices (/dev/vdb). mkfs, e2fsck and resize2fs operate on partitions (/dev/vdb1). Getting it backwards in either direction destroys something.
  • The tools will not save you. sgdisk -g /dev/vdb1 is a syntactically valid request that the tool has no way to know is a mistake. There is no confirmation prompt for "are you sure this is a whole disk".
  • Past 2 TiB, MBR is not an option — plan the GPT conversion as part of the expansion, not as a surprise afterwards.
  • Long-running recovery commands belong in tmux. An e2fsck killed halfway through by a dropped connection can leave you worse off than when you started.
  • A weird number in df is a metadata problem, not a data problem. 64 ZB, negative sizes, absurd inode counts — these mean a structure was misread, and misread structures are frequently recoverable. Do not conclude the data is gone.
