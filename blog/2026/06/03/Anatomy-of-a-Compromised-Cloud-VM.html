Anatomy of a Compromised Cloud VM

Posted on June 3, 2026 ()

Categories: security  incident response  forensics 

The ticket looked mundane. A customer could not log into their cloud virtual machine — SSH timed out, and the VNC console came up blank. No configuration had been changed on their side, they said, and could we please take a look from the platform.

The instinct in that situation is to check the obvious things: security group rules, the network path, whether sshd is running, whether the disk is full. All of those turned out to be fine, and that is what made the case interesting. The VM was not down. It was extremely busy — just not with anything the customer had asked it to do.

What follows is the evidence chain we assembled, roughly in the order we found it. I have anonymised the addresses and defanged the domains, but the shapes and the log lines are real. The point of writing it up is that no single item below is proof of anything. It is the combination that makes the conclusion unavoidable.

Step 1: the monitoring graphs disagree with each other

The first genuinely suspicious signal came from comparing two sources of monitoring data that normally agree.

The guest-side agent — the little daemon inside the VM that reports CPU, memory and disk metrics — stopped reporting at 05:55 on the day of the incident. Its graphs flat-line from that moment. On its own that just looks like a dead agent.

The platform-side counters, which are collected outside the guest on the virtualisation layer and cannot be tampered with from inside, told a completely different story. Both private and public network traffic were still flowing. The number of active outbound TCP connections had jumped sharply and stayed high.

That contradiction is the whole case in miniature:

When the in-guest agent goes silent but the hypervisor still sees heavy traffic, the machine has not crashed. It has stopped being under the owner's control.

A dead VM produces no traffic. A healthy VM produces traffic and metrics. A VM producing traffic but no metrics is one where something has killed or starved the monitoring agent while continuing to use the network. That is worth escalating on its own.

The shape of the outbound connection count mattered too. Normal workloads ramp. This went from a flat baseline to a plateau in a couple of minutes and stayed there — the signature of a program that opens connections in a loop as fast as the kernel will let it.

Step 2: DNS tells you what the malware wants

With platform-side packet capture running on the VM's virtual NIC, DNS was the obvious place to start. There are two good reasons for that, and they are worth internalising for any incident:

  • Almost everything needs DNS. Command-and-control check-ins, payload downloads, mining pool endpoints, data exfiltration targets — nearly all of it starts with a name lookup.
  • DNS is in clear text. Even when the payload traffic that follows is TLS-encrypted and opaque, the query that preceded it usually is not. DNS is the cheapest intelligence in the capture.

Every query in the capture originated from the VM itself. Three patterns stood out.

Pattern 1: domains that are not trying very hard to hide

VM → resolver-A   A     mushhacker[.]goat666[.]xyz
VM → resolver-A   AAAA  mushhacker[.]goat666[.]xyz
resolver-A → VM   A     111[.]207[.]241[.]73

VM → resolver-B   A     pool[.]supportxml[.]com
VM → resolver-B   AAAA  pool[.]supportxml[.]com
resolver-B → VM   A     142[.]248[.]80[.]25

The first domain contains the word "hacker" in the hostname, sits on a .xyz TLD — cheap and heavily used for short-lived infrastructure — and its AAAA record returns an SOA pointing at a dynamic-DNS provider. That last detail is a tell in itself: dynamic DNS lets an operator move infrastructure between IPs without re-registering anything.

The second is more careful. supportxml.com is built to look like some legitimate support or API service, and skimming a log at speed you might well let it past. What gives it away is the pool label — the standard naming for mining pools, proxy pools and task pools — combined with the query pattern.

That query pattern is the part I would encourage you to look for more than any specific string. A and AAAA fired back-to-back, in the same millisecond, with no other traffic in between. A browser does not do that. A human does not do that. A resolver library called directly from a program in a tight loop does exactly that. The cadence is the indicator of compromise; the domain name is just the label on it.

Pattern 2: a burst of "what is my IP" services

Within a couple of seconds, the same host queried nine different public IP echo services:

api.ipify.org          ipv4.icanhazip.com     ipecho.net
wtfismyip.com          ipinfo.io              checkip.amazonaws.com
api4.ipify.org         api.myip.com           ip.seeip.org

Each of these is entirely benign. They are ordinary free services that return the public IP address your traffic appears to come from, and plenty of legitimate software uses one of them.

What is not benign is nine of them from one host inside a few seconds, spread across several different resolvers. No human and no normal application fans out like that. The behaviour has a specific purpose:

  • Find the real egress IP. Behind cloud NAT, the address on the interface is not the address the outside world sees. Malware that wants to report a usable target, or to check whether its traffic is being proxied, has to ask someone outside.
  • Fingerprint the environment. The answer reveals the hosting provider and region, which drives what the next stage does — mining is worth it on some instance types and not others, and some operators skip certain providers entirely.
  • Check in. Many families phone home with the egress IP as part of the initial registration with their controller.

The redundancy is deliberate engineering on the attacker's part. Any single one of these services might be blocked, rate-limited or dead, so the malware tries a pile of them and takes the first answer.

This makes a genuinely useful detection rule, and it is one you can deploy without knowing a single attacker domain in advance:

Three or more distinct public-IP-echo services queried by the same host inside a short window is automated environment fingerprinting. Alert on it even though every individual domain is legitimate.

Pattern 3: several resolvers in parallel

The queries were not all going to the resolver the VM was configured to use. Four different resolvers appeared in the capture, all receiving the same kind of traffic at the same cadence, including well-known public DNS services that the guest's /etc/resolv.conf never mentioned.

This means the malware ships its own resolver list rather than using the system configuration. Two benefits for the attacker: resolution keeps working if the environment's DNS is broken or restrictive, and it sidesteps any DNS filtering, sinkholing or RPZ policy applied at the configured resolver. For a defender it is a strong signal in its own right — a process talking directly to hard-coded nameservers is bypassing your controls on purpose, and that is not something well-behaved software does.

It also has a practical consequence worth planning for: if your entire DNS-based blocking strategy lives on the resolver you hand out via DHCP, malware like this never sees it. Blocking outbound UDP/53 and TCP/53 to everything except your own resolvers is what actually makes DNS policy enforceable.

Step 3: the packet capture shows what it is doing

Beyond DNS, the capture was dominated by one thing: a very large number of short-lived TCP connections from the VM to port 22 on a long list of unrelated public addresses.

Every flow had the same shape:

  • Source: the customer's VM, on every single flow.
  • Destination: hundreds of unique public IPs with no business relationship to the customer, in no obvious network order.
  • Destination port: 22, without exception.
  • Source port: a different ephemeral high port each time — the classic scanner fingerprint, since each attempt is a fresh socket.
  • TCP flags: a flood of SYNs, with mostly RST or RST/ACK coming back. Most targets were refusing.
  • Cadence: bursty, machine-paced, no pauses that look like a human at a keyboard.

The direction is what matters here. Almost every SSH-related alert an operations team sees is inbound — somebody on the internet brute-forcing your port 22, which is background radiation and usually ignorable. This was the mirror image. The VM was the SSH client, fanning out to the world.

There is no legitimate workload that behaves like that. Configuration management tools reach a known inventory, not hundreds of random addresses. Backup jobs have fixed destinations. An automated SSH client with an unbounded, unrelated target list is a scanner or a brute-forcer, full stop.

This also finally explained the original symptom. The customer could not log in because:

  • the kernel's connection tracking table was completely exhausted, so new connections — including the customer's inbound SSH — were being dropped before they reached sshd;
  • CPU and memory were consumed by the scanner process;
  • sshd and the console agent were starved and timing out;
  • the kernel was dropping packets under load regardless of what they were.

The login failure was never the problem. It was a side effect.

Step 4: the serial console log names the culprit

The serial console log is the most underrated artefact in cloud incident response. It is captured outside the guest, so it survives when the guest filesystem is unreliable, when journald has been killed, and when an attacker has cleaned up after themselves inside the machine. Get it early.

Six patterns appeared together.

1. Connection tracking table exhausted

[05:45:41] nf_conntrack: table full, dropping packet
[05:45:41] nf_conntrack: table full, dropping packet
[05:45:41] nf_conntrack: table full, dropping packet
...
[05:45:46] net_ratelimit: 5367 callbacks suppressed
[05:45:46] nf_conntrack: table full, dropping packet

Every tracked connection occupies an entry in nf_conntrack, and the table has a fixed size. Open connections faster than they retire and it fills, at which point the kernel simply drops packets it cannot track.

The second line is the one people skip over, and it is the important one. net_ratelimit: 5367 callbacks suppressed means the kernel discarded 5,367 additional identical messages in that interval because it was printing too fast. What you can see in the log is a small fraction of what actually happened. When you find this line, mentally multiply everything nearby.

Taken alone, a full conntrack table proves nothing — a busy proxy or a load balancer can fill it during normal operation. It is alarming because of what filled it.

Two strings to grep for when you want to check this quickly: table full, dropping packet and callbacks suppressed. Both in a tight window means a connection storm is in progress.

2. Remote SSH servers replying to us

[UFW BLOCK] SRC=203.0.113.181  DST=<vm>  PROTO=TCP  SPT=22  DPT=51944  ACK SYN
[UFW BLOCK] SRC=203.0.113.57   DST=<vm>  PROTO=TCP  SPT=22  DPT=58324  ACK SYN
[UFW BLOCK] SRC=198.51.100.157 DST=<vm>  PROTO=TCP  SPT=35606 DPT=35606 ACK
[UFW BLOCK] SRC=198.51.100.43  DST=<vm>  PROTO=TCP  SPT=22  DPT=40986 ACK SYN

Read the ports carefully, because the direction is easy to get backwards. Source port 22, destination port a high ephemeral port. These are not people attacking the VM on SSH. These are SSH servers around the internet completing the handshake for connections the VM opened, and the host firewall blocking the replies on the way back in.

Dozens of distinct remote addresses appear across these lines. A machine that receives SYN/ACKs on port 22 from dozens of strangers has been sending SYNs to dozens of strangers.

3. ICMP messages carrying our own packets back

This was the single most conclusive line in the whole incident:

[UFW BLOCK] SRC=203.0.113.251  DST=<vm>  PROTO=ICMP  TYPE=3  CODE=3
   [SRC=<vm>  DST=10.0.2.22  PROTO=TCP  SPT=41196  DPT=22  SYN]
[UFW BLOCK] SRC=198.51.100.188 DST=<vm>  PROTO=ICMP  TYPE=3  CODE=3
   [SRC=<vm>  DST=198.51.100.188  PROTO=TCP  SPT=52838  DPT=22  ACK RST]

ICMP type 3 code 3 is "destination unreachable, port unreachable". By design, an ICMP error message quotes the header of the packet that triggered it, so the router or host can tell you exactly what it could not deliver.

That quoted header, in brackets, is the VM's own outbound packet, preserved verbatim: source is the VM, destination port 22, SYN flag set. This is not inference from a pattern or a statistical argument. It is the machine's outgoing SSH connection attempt, returned by the network with a note saying nobody was listening. You cannot explain it away.

4. An unknown process, repeatedly killed for memory

Out of memory: Killed process 3881105 (cx)
   total-vm:13514232kB  anon-rss:11836872kB  UID:1000
Out of memory: Killed process 3882282 (cx)
   total-vm:10006124kB  anon-rss:8723848kB   UID:1000
Out of memory: Killed process 3885623 (cx)
   total-vm:9940356kB   anon-rss:8685648kB   UID:0
... repeating every 30-90 minutes

A process named cx. Four observations, each independently suspicious:

  • The name. Two characters, meaningless, memorable to nobody. Dropper-staged binaries are routinely named like this precisely because a short name blends into a ps listing.
  • The footprint. Eight to twelve gigabytes of anonymous RSS per run. This is not a helper script; it is holding an enormous working set, consistent with a large target list and many concurrent connection attempts.
  • The UID change. This is the important one. Early kills show UID:1000 — an ordinary user account. Later ones show UID:0. The same program is now running as root. A privilege escalation happened between those two log lines, which changes the scope of the incident entirely: from that point on nothing in the guest can be trusted, including any evidence stored inside it.
  • The persistence. The OOM killer terminates it and it comes straight back, on a regular cadence. Something is deliberately restarting it.

5. A service that says what it is

systemd[1]: ssh-scanner.service: Main process exited,
            code=killed, status=9/KILL
systemd[1]: ssh-scanner.service: Failed with result 'signal'.

A systemd unit named ssh-scanner. Whoever built this tooling was not concerned about being discovered, which is itself informative — it suggests commodity malware run at scale rather than a targeted intrusion trying to stay quiet.

It also answers the "what is restarting cx" question. The crash timestamps line up one-to-one with the OOM kills: the OOM killer takes down cx, the unit fails, systemd restarts it, and the cycle repeats every half hour or so. That closes the loop between the packet capture and the process table.

6. Journald itself falls over

systemd[1]: systemd-journald.service: Main process exited,
            code=dumped, status=6/ABRT
systemd[1]: systemd-journald.service: Failed with result 'watchdog'.
systemd-journald: File /var/log/journal/.../system.journal corrupted
                  or uncleanly shut down, renaming and replacing.

The logging daemon missed its watchdog deadline under the load and was aborted, and journal files were corrupted in the process. This has an immediate practical consequence for anyone working an incident like this: the in-guest logs are now incomplete and partly unreadable, and they got that way on their own. Not through anti-forensics — just through load. It is a good argument for shipping logs off the instance continuously, and for reaching for the serial console early rather than after you have exhausted everything inside the guest.

The pattern worth keeping

The reusable output of this incident is not the domains or the IPs, which will be dead within weeks. It is the combination. When all of the following show up together in a serial console log, treat the guest as compromised until proven otherwise:

  1. nf_conntrack: table full together with net_ratelimit: callbacks suppressed.
  2. Many firewall-block lines with PROTO=TCP SPT=22 DPT=<high port> — replies to connections you initiated.
  3. ICMP type 3 messages whose quoted payload shows SRC=<local IP> DPT=22 SYN.
  4. An unfamiliar, short-named process repeatedly OOM-killed, especially if its UID changes to 0 partway through.
  5. A systemd unit whose name contains scanner, scan, brute or miner.
  6. systemd-journald watchdog aborts and corrupted journal files.

Any one of these has an innocent explanation. All six at once do not.

Recovery

Once a guest has been rooted, there is no cleaning it. You do not know what was installed, what was modified, or what will run at next boot, and the effort of finding out reliably exceeds the effort of rebuilding. The sequence we used:

  1. Boot the disk in rescue mode, so nothing from the compromised system executes.
  2. Copy out only the data. Documents, databases, configuration you have reviewed. Never binaries, never executables, never anything from /usr/bin, and be careful with anything that runs at startup — cron entries, systemd units, shell profiles.
  3. Reinstall from a known-clean image. Not a snapshot from after the compromise, which may already contain the implant.
  4. Restore the data onto the fresh system.
  5. Harden before exposing it, because whatever let them in the first time is still true.

That last step is where the actual value is, and it splits into three areas.

Network exposure

  • Restrict inbound TCP/22 to specific trusted source addresses. A default rule allowing SSH from 0.0.0.0/0 is the single most common way machines like this get taken.
  • Put administrative access behind a bastion host or VPN rather than opening management ports to the internet.
  • Bind databases, dashboards and admin interfaces to private subnets only.
  • Block outbound TCP/22 to the public internet. This one is rarely done and would have stopped this incident dead. A normal application server has no reason to initiate SSH connections to arbitrary hosts. The scanner would have been useless from the first minute.
  • Restrict outbound DNS to your own resolvers, and alert on volume spikes. Both the domain lookups and the IP-echo burst would have been visible or blocked.

Authentication

  • Set PasswordAuthentication no. Key-based authentication removes the entire brute-force attack surface.
  • Use ed25519 or RSA-4096 keys, one per user, and revoke them when people leave.
  • Set PermitRootLogin no.
  • If passwords must remain for some reason: sixteen characters minimum, and an account lockout policy so an attacker gets a handful of attempts rather than millions.
  • Add multi-factor authentication at the bastion or VPN layer for human access.

Backup and detection

  • Schedule disk snapshots and keep at least one copy off the instance. Recovery here was straightforward because the data was recoverable.
  • Test a restore quarterly. An untested backup is a hypothesis, not a backup.
  • Alarm on outbound connection count, not just CPU and memory. The outbound TCP spike was visible from minute one and was the earliest reliable signal available.
  • Run a host security agent and actually triage what it reports.
  • Ship logs off the instance continuously, and remember the serial console — it is outside the guest and survives everything that happens inside it.

The theme running through all of it: the strongest evidence came from outside the guest — platform counters, packet capture on the virtual NIC, serial console output. Everything inside the machine was either killed, starved or corrupted. Design your monitoring on the assumption that one day the guest will be lying to you.
