Why RTC Time Changes Do Not Survive a Reboot on a Cloud VM

Posted on March 18, 2026 ()

Categories: virtualization  timekeeping  linux 

A customer wanted the hardware clock on their cloud VMs set to local time rather than UTC. Their application read the RTC directly and expected local time, and porting it was not on the table.

This is a well-documented operation. Set the hardware clock, tell the system it is in local time, done:

$ sudo timedatectl set-local-rtc 1
$ sudo hwclock --systohc --localtime

They did that over SSH. It worked. They rebooted, and the RTC was back to UTC.

They tried it again through the VNC console instead of SSH, using the same commands. That time it persisted.

Same commands, same image, same instance — different result depending on how you were logged in. That is a strange enough claim that our first assumption was a procedural difference, so we set out to test it properly.

Ruling out the obvious

Before believing something that surprising, it is worth eliminating the alternatives. We ran the same sequence across a matrix of operating systems and instance types:

Guest OSInstance types testedOver SSHOver VNC
CentOS 7.9six families, Intel and AMDLost on rebootPersists
Ubuntu 22.04six families, Intel and AMDLost on rebootPersists
Vendor Linux 3.2six families, Intel and AMDLost on rebootPersists

Completely consistent. Not OS-specific, not instance-type-specific, not a one-off. The only variable that changed the outcome was the access path.

Which is genuinely counter-intuitive, because SSH and VNC should not differ in any way that matters here. Both give you a shell. Both run the command as root. The kernel does not know or care which TTY a process is attached to when it services an ioctl.

So the difference is not in how the command runs. It is in what happens to the result afterwards.

A virtual machine has no battery

On a physical machine, the RTC is a small piece of dedicated hardware with its own battery. It keeps time while the machine is powered off, and writing to it changes a physical device that persists across power cycles by construction.

A virtual machine has none of that. The "RTC" the guest sees is emulated by the hypervisor. When the guest writes to it, the write goes into a data structure belonging to the virtualisation layer, and what happens to that structure — whether it is kept, and for how long — is a decision made outside the guest.

This is the crux. In a VM, the persistence of a hardware clock write is a hypervisor policy question, not a guest question. The guest can issue the write successfully, read it back successfully, and still have it discarded, because the guest is not the thing that decides.

On a reboot the guest's emulated RTC is typically re-initialised from the host — from the host's clock, in UTC, which is what essentially every hypervisor and cloud platform standardises on. Any local modification is simply overwritten, and the guest gets no indication that this happened.

Why the console behaves differently

The console is part of the virtualisation stack. Unlike SSH — which is an ordinary network service running inside the guest, with no relationship to the hypervisor at all — the VNC console is provided by the platform, and the session runs through machinery on the host side.

The practical upshot is that the console path is bound up with the platform's own device model in a way that a network login is not, and in this environment a console-initiated change landed somewhere the reinitialisation did not clobber. A change made over SSH sat purely in guest-visible emulated state, which is exactly what gets reset.

I want to be careful about how strongly to state this, because we characterised the behaviour without access to the hypervisor internals that would let us describe the mechanism precisely. The observation is solid and reproduced across eighteen combinations. The explanation is our best reading of it.

But the general lesson holds regardless of the details, and it is the part worth carrying to other platforms:

In a virtual machine, the emulated RTC is owned by the hypervisor. A guest can write to it, but the guest does not decide whether that write survives a power cycle.

What to do instead

Relying on console-specific behaviour to make a setting stick is not a foundation to build on. It is undocumented, it is platform-specific, and it will not survive a live migration or a platform upgrade. Better approaches, in order of preference:

Set the timezone, not the hardware clock. This is what the customer actually needed in almost every case:

$ sudo timedatectl set-timezone Asia/Shanghai

The system clock stays on UTC internally — which is what you want, since it makes log correlation and daylight-saving transitions sane — and everything that displays or formats a time shows local time. Applications using standard time functions get local time automatically. No hardware clock involved, and it persists because it is a file in the guest's own filesystem.

Re-apply at boot if you truly need a local RTC. If an application really does read the RTC directly and cannot be changed, stop trying to make the setting persist and just apply it on every boot:

[Unit]
Description=Set RTC to local time
After=systemd-timesyncd.service

[Service]
Type=oneshot
ExecStart=/usr/bin/timedatectl set-local-rtc 1

[Install]
WantedBy=multi-user.target

This is honest about what is happening: the hypervisor resets the clock at boot, so you set it again after boot. It works on every platform because it does not depend on any platform-specific behaviour.

Run NTP and treat the RTC as unreliable. Which is the right mental model for a VM in general.

The broader point: VM clocks drift

This case is one instance of a wider truth that is worth being explicit about, because it comes up constantly in cloud environments and is often misdiagnosed as an application bug.

Timekeeping on physical hardware relies on counting something that ticks at a known rate — usually the TSC, measured once at boot. Virtualisation breaks several of the assumptions that makes that work:

  • The guest does not run continuously. It is descheduled whenever the host has other work. From inside, time appears to jump.
  • Live migration changes the hardware underneath you. The new host's TSC has a different value and possibly a different frequency. The value the guest calibrated at boot is now wrong.
  • Suspend and resume produce large jumps — potentially hours, if a VM is resumed long after it was suspended.

Hypervisors mitigate this with paravirtualised clocks — pvclock for KVM and Xen guests, the TSC reference page for Hyper-V — which give the guest a stable, fixed-frequency time source that the host keeps meaningful across migration. Modern guests use these automatically and the situation is much better than it was a decade ago.

It is still not perfect, and one practical detail catches people out: a jump large enough will make ntpd give up. By default it refuses to step the clock beyond a threshold, on the reasonable assumption that such a discrepancy indicates a fault rather than something to silently correct. In a VM it is routine. The tinker panic 0 directive tells ntpd to step the clock regardless, and it is standard practice in virtualised guests. chrony, which is the better default on modern systems, handles this natively with makestep and generally needs less coaxing.

Takeaways

  • The RTC in a VM is emulated, and the hypervisor owns it. A successful write from the guest does not mean a durable write.
  • Prefer the timezone to the hardware clock. timedatectl set-timezone solves the actual requirement in almost every case, and it persists because it is guest-owned state.
  • If you need a setting after every boot, apply it after every boot. A small systemd unit is more reliable than any mechanism that depends on hypervisor behaviour you cannot see.
  • Run NTP or chrony, and expect jumps. Configure the daemon to step the clock rather than give up.
  • When identical commands give different results, look outside the guest. The variable that mattered here was not in the operating system at all.
