Moving Millions of Small Files Between Cloud VMs

Posted on March 25, 2026 ()

Categories: storage  linux  performance 

A customer's virtual machine would not boot. The data on its disk was intact and reachable, so the recovery plan was straightforward: attach the disk to a working instance, copy the data to a replacement VM, and rebuild.

The data was 188 GB. That is not a large amount by modern standards — on a decent cloud network it should move in well under an hour.

It was spread across millions of small files: a user home directory tree, source files, thumbnails, caches, per-project scratch data. That changes the problem completely, and the first approach we tried made it worse rather than better.

Why small files are a different problem

The intuition that 188 GB is a bandwidth problem is wrong here. Moving a file is not just moving its bytes. For each one, both sides must:

  • look up or create a directory entry;
  • allocate or read an inode;
  • open, read or write, and close;
  • apply ownership, permissions and timestamps.

That is a handful of syscalls and several small, scattered I/O operations per file, and none of it scales with file size. Ten million files at ten operations each is a hundred million operations regardless of whether the files are 2 KB or 2 MB.

The consequence: the transfer is limited by IOPS and syscall overhead, not by bandwidth. A cloud disk rated at, say, 20,000 IOPS gives you a hard ceiling on how many files per second you can touch, and no amount of network capacity moves it. It is common to see a link sitting at a few percent utilisation while the copy crawls.

Latency compounds it. Anything that does a round trip per file — scp in a loop, rsync over a slow link, a naive recursive copy — pays the round-trip time millions of times.

The first plan, and why it was wrong

The instinct was: archive everything into one big file, transfer that at full speed, unpack on the far side. One large sequential stream instead of millions of tiny random ones — exactly the shape a network wants.

$ tar -czf /mnt/backup/data.tar.gz /mnt/vm1/data/users

It is a reasonable-sounding plan and it has three problems.

You need somewhere to put it. A 188 GB archive requires 188 GB of free space, which means provisioning another volume, attaching it, and paying for it.

You do the I/O twice. Every byte is read from the source disk and written to the archive, then read from the archive and written across the network, then written again on the far side. The expensive part — millions of small reads on the source — still happens in full, and now there is a large sequential write and read added to it.

Nothing overlaps. This is the real cost. The archive step must finish completely before the transfer starts, and the transfer must finish before unpacking begins. Three long phases in strict sequence, with the network idle during the first and third. Hours of elapsed time where one resource is saturated and the others do nothing.

Compression does not rescue it either. Much of the data was already compressed — images and media — so gzip burned CPU for very little gain, on a single core, while everything else waited.

The plan that worked

The insight is that the archive is a means, not an end. We want the stream — the sequential representation of the tree — but we do not need it to exist as a file anywhere. So pipe it directly from tar on one side into tar on the other:

$ tmux new -s transfer

$ tar -cf - -C /mnt/vm1/data/users . \
    | pv -s 188G \
    | ssh -C user@10.0.1.20 "tar -xf - -C /data/users"

Reading it left to right:

  • tar -cf - writes the archive to standard output rather than a file.
  • -C /mnt/vm1/data/users . changes directory first and archives ., so paths in the archive are relative. This matters: it lets you extract into a different directory on the far side, and avoids the leading-slash awkwardness of absolute paths.
  • pv -s 188G passes the stream through unchanged while showing throughput, elapsed time and an ETA. With -s supplying the expected size, you get a genuine progress bar — worth a great deal on a multi-hour job where the alternative is staring at a cursor.
  • ssh -C carries the stream to the target, compressing in transit.
  • tar -xf - -C /data/users extracts from standard input directly into place on the far side.

No intermediate file exists at any point. No extra disk is provisioned. And crucially, every stage runs concurrently: the source disk is reading while the network is transmitting while the target disk is writing. The pipeline runs at the speed of its slowest stage rather than the sum of three sequential phases.

The other benefit is that the per-file overhead is paid once, on the source, in a single sequential walk of the tree. There is no per-file network round trip at all — from the network's point of view this is one long stream.

Run it in tmux

The tmux new -s transfer on the first line is not decoration. This job runs for hours, and if it is attached to your SSH session then a dropped connection kills it — after which you get to work out how much was transferred and how to resume, which for a tar stream means largely starting over.

Inside tmux the pipeline belongs to the tmux server, not your terminal. Disconnect, reconnect later, tmux attach -t transfer, and the progress bar is still there.

This applies to any long-running operation on a remote machine — filesystem checks, database imports, large builds. It is the cheapest reliability improvement available and it costs seven characters.

About ssh -C

Compression in SSH is worth thinking about rather than always enabling.

Use it when the data compresses well — source code, logs, text, uncompressed databases — and when the network is the bottleneck. Trading CPU for bytes on the wire is a good deal when you have CPU to spare.

Skip it when the data is already compressed, which was largely the case here. You pay full CPU cost for a few percent reduction, and on a fast internal network the compression itself can become the limiting factor — ssh -C is single-threaded, so one core has to keep up with the whole stream.

The empirical approach: run it for thirty seconds with -C, watch the rate in pv, run it without, compare. On a fast link with pre-compressed data, dropping -C often makes it faster.

The storage tier matters more than you expect

One thing this exercise made very clear: for small-file work, the disk performance tier is the dominant variable.

Cloud block storage is sold in tiers whose IOPS ratings differ by an order of magnitude or more. For a large sequential transfer the tier barely matters — even modest volumes saturate a typical network link on sequential reads. For millions of small files, IOPS is the transfer rate, and the difference between a basic volume and a high-performance one can be the difference between three hours and most of a day.

Two practical consequences:

  • Check both ends. The target's write IOPS limits you just as surely as the source's read IOPS, and it is easy to focus on the source and forget that the destination is also doing millions of small operations.
  • Consider a temporary upgrade. Most cloud providers let you raise a volume's performance tier without recreating it. Bumping it for the duration of a migration and lowering it afterwards is a legitimate and frequently overlooked tactic — you are paying for a few hours of high performance to save most of a day of waiting.

Variations worth knowing

If you need to resume, use rsync instead. It is slower per file because it builds a file list and compares, but an interrupted run picks up where it left off:

$ rsync -aHAX --info=progress2 /mnt/vm1/data/users/ user@10.0.1.20:/data/users/

The flags matter for a faithful copy: -H preserves hard links, -A ACLs, -X extended attributes. Plain -a silently drops all three.

If one stream cannot saturate the link, split the work by top-level directory and run several pipelines at once. With separate tmux windows this is easy to manage, and it lets multiple cores handle SSH encryption in parallel — often the real ceiling on a fast network:

$ for d in a b c d; do
    tmux new-window -d "tar -cf - -C /src/$d . | ssh user@10.0.1.20 'tar -xf - -C /dst/$d'"
  done

If both machines are on a fast, trusted private network, SSH's encryption overhead can dominate. netcat removes it entirely — receiver first, then sender:

# on the target
$ nc -l -p 9000 | tar -xf - -C /data/users

# on the source
$ tar -cf - -C /mnt/vm1/data/users . | pv -s 188G | nc 10.0.1.20 9000

This is unencrypted and unauthenticated. Private network between hosts you control only.

Summary

  • Millions of small files is an IOPS problem, not a bandwidth problem. Optimise the number of operations, not the size of the pipe.
  • Do not stage an archive you do not need. Piping tar into tar over SSH overlaps read, transfer and write, and needs no scratch space.
  • Use tmux for anything that runs for hours.
  • Compression is a judgement call. Measure it; do not assume -C helps.
  • Check the disk tier at both ends, and consider raising it temporarily for the duration of a migration.
  • pv is worth installing. Knowing whether a job has two hours or twelve left changes what you do next.
